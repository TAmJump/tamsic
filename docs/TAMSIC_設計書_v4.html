サービスの 3 層構造

Layer 1（公開サイト）: tamsic.tamjump.com — 誰でもアクセス可。非会員には LOCKED / EARLY ACCESS の壁を見せて登録へ誘導。
Layer 2（会員ラウンジ）: My Page — Cognito 認証後のみ。先行視聴中の曲・残高・購入履歴を集約。
Layer 3（フル試聴）: コイン消費でロック解除。1 回試聴のみ、歌詞は保護モジュール経由で表示。

ブランド原則

  • 欲求で動かす。説明で登録させない。触った瞬間に壁を置いて、その壁が「欲しい」を立ち上げる。
  • 会員登録は入口の壁、コイン課金は体験の拡張。二層で収益構造を組む。
  • 展示サイトではなく稼ぐ構造のサイト。世界観の美しさは維持しつつ、導線は妥協しない。
  • 触れそうで触れない状態を作れているか。これが全ページ共通のレビュー観点。

コイン書き込みの仕組み

技術メモ: ブラウザから直接 Cognito の UpdateUserAttributes API を fetch で呼ぶと CORS でブロックされる。そのため amazon-cognito-identity-js SDK の cognitoUser.updateAttributes() を使う。SDK は CORS を回避する処理を内蔵している。 
// auth.js の _updateUserAttributes() の仕組み
const cognitoUser = _getCognitoUser();
const attributeList = [
  new AmazonCognitoIdentity.CognitoUserAttribute({
    Name: 'custom:coins', Value: '500'
  })
];
cognitoUser.updateAttributes(attributeList, callback);

新規登録直後は custom:coins が未設定（null）のため、コード側で parseInt(null) → 0 として扱う。初回購入反映時に Cognito に書き込まれる。

10歌詞保護モジュール
前提: Web 上で「絶対に複製不可」は技術的に不可能。OS レベルのスクリーンショット、外部カメラ撮影は完全には防げない。v4 の目的は (1) 容易には持ち出せない水準までハードルを上げる (2) 流出時に追跡できる透かしを埋め込む、の 2 点。

保護の 5 層（lyrics-guard.js）

層技法防げる範囲
CSSuser-select:none, -webkit-touch-callout:none, -webkit-user-drag:noneテキスト選択・モバイル長押し・ドラッグ
イベントcontextmenu / copy / cut / selectstart / dragstart / beforecopy を preventDefault右クリックメニュー・コピー・ドラッグ
キーCtrl/⌘+C, Ctrl+A, Ctrl+S, Ctrl+P, PrintScreen を抑止ショートカットでのコピー・保存・印刷
透かしメールアドレス＋日付の SHA-256 ハッシュを歌詞背面に 5% 不透明で繰り返し描画流出元の特定・心理的抑止
難読化歌詞テキストを 1 文字 1 span に分割。DOM 上で連続文字列として取れない単純な DOM スクレイピング

  onPrintScreen: 'blur'            // 'blur' | 'hide' | null

追加機能

  • DevTools 検知: window.outerHeight - window.innerHeight が 160px を超えたら歌詞を blur。
  • PrintScreen 検知: Windows で PrintScreen キーを検知したら歌詞を 3 秒間 blur。同時にクリップボードのクリアを試行。

やらないこと（v4 では見送り）

  • Canvas / SVG レンダリング: 選択・コピーを完全に防げるが、アクセシビリティと検索性が落ちる。
  • 歌詞の画像化: スクレイピングに弱く、OCR で簡単に抜ける。
  • DRM（Widevine など）: 歌詞テキストには過剰。音声側も現時点では導入しない。

流出時の対応

  1. 流出している歌詞テキスト上の透かしハッシュを画像解析ツールで読み取る
  2. 内部マッピングテーブル（SHA-256 ハッシュ → 会員 email）で逆引き
  3. Cognito 側で当該ユーザーを AdminDisableUser で停止
  4. lyrics-guard.js のアップデートを検討

// release-control.js 冒頭
const ADMIN_EMAILS = ["animalb001@gmail.com"];

function _isAdmin() {
  try {
    if (typeof getUserInfo === "function") {
      const u = getUserInfo();
      if (u && u.email &&
          ADMIN_EMAILS.map(e => e.toLowerCase()).includes(u.email.toLowerCase()))
        return true;
    }
  } catch(e) {}
  return false;
}

function getState(title) {
  const t = config.tracks[title];
  if (!t) return "full";
  if (_isAdmin()) return "full"; // ← ここで全曲解放
  // ...通常の日付判定
}

21セキュリティ監査・運用ルール

静的サイトの構造的制約

前提: TAMSIC は GitHub Pages 上の完全静的サイトのため、サーバー側アクセス制御が存在しない。repo にコミット済みの MP3 は URL を知っている人なら誰でもアクセスできる。これは仕様上の制約であり、対応策は運用側で決めている。

対応方針（v4 採用）

以下の 3 層防御で緩和している:

  1. 未公開 MP3 は repo にコミットしない（Vault で保管）— 404 で防御
  2. サイト側で release-control.js + getTrackAudioSrc() の二重ゲート— UI からは locked/preview 曲を再生不可能
  3. GitHub Actions が公開日に自動配置— 人間の手動作業を排除し運用ミスを減らす

2026.04.23 発生事案: アーティストページの構造が揃っていなかった

2026.04.23 発生事案: 別アーティストの同名曲が add-track.js で追加できない

24他サービスとの統一事項
TAmJ.Corp が運営する他サービス（Touchvoo・Medvoo・Adavoo）と、運用ルールを揃える。業種は違うが、ログイン・課金・パスワード管理・設計書フォーマットは統一することで引き継ぎコストが下がる。

Touchvoo からの移植項目

項目Touchvoo の実装TAMSIC v4 での扱い
パスワードハッシュSHA-256 で D1 に保存Cognito 管理のためハッシュ化は不要。生 PW を保持しないのは共通
パスワードリセット管理者から再発行可能（account-master）Cognito の forgotPassword 機能を使用。管理者からも AWS コンソールで一時 PW 発行可
Square 決済リンク月額サブスク型（¥200/月）都度課金型（coin パック）。Webhook の将来実装パターンだけ流用

統一できない項目（あえて分ける）

項目理由
データベース層Touchvoo は Cloudflare D1 / TAMSIC は Cognito カスタム属性 + localStorage。規模が違うため統一不要。
