サービスの 3 層構造

Layer 1（公開サイト）: tamsic.tamjump.com — 誰でもアクセス可。非会員には LOCKED / EARLY ACCESS の壁を見せて登録へ誘導。
Layer 2（会員ラウンジ）: My Page — Cognito 認証後のみ。先行視聴中の曲・残高・購入履歴を集約。
Layer 3（フル試聴）: コイン消費でロック解除。1 回試聴のみ、歌詞は保護モジュール経由で表示。

ブランド原則

  • 欲求で動かす。説明で登録させない。触った瞬間に壁を置いて、その壁が「欲しい」を立ち上げる。
  • 会員登録は入口の壁、コイン課金は体験の拡張。二層で収益構造を組む。
  • 展示サイトではなく稼ぐ構造のサイト。世界観の美しさは維持しつつ、導線は妥協しない。
  • 触れそうで触れない状態を作れているか。これが全ページ共通のレビュー観点。

コイン書き込みの仕組み

技術メモ: ブラウザから直接 Cognito の UpdateUserAttributes API を fetch で呼ぶと CORS でブロックされる。そのため amazon-cognito-identity-js SDK の cognitoUser.updateAttributes() を使う。SDK は CORS を回避する処理を内蔵している。 
// auth.js の _updateUserAttributes() の仕組み
const cognitoUser = _getCognitoUser();
const attributeList = [
  new AmazonCognitoIdentity.CognitoUserAttribute({
    Name: 'custom:coins', Value: '500'
  })
];
cognitoUser.updateAttributes(attributeList, callback);

新規ユーザーの初期値

新規登録直後は custom:coins が未設定（null）のため、コード側で parseInt(null) → 0 として扱う。初回購入反映時に Cognito に書き込まれる。

10歌詞保護モジュール
前提: Web 上で「絶対に複製不可」は技術的に不可能。OS レベルのスクリーンショット、外部カメラ撮影は完全には防げない。v4 の目的は (1) 容易には持ち出せない水準までハードルを上げる (2) 流出時に追跡できる透かしを埋め込む、の 2 点。

保護の 5 層（lyrics-guard.js）

層技法防げる範囲
CSSuser-select:none, -webkit-touch-callout:none, -webkit-user-drag:noneテキスト選択・モバイル長押し・ドラッグ
イベントcontextmenu / copy / cut / selectstart / dragstart / beforecopy を preventDefault右クリックメニュー・コピー・ドラッグ
キーCtrl/⌘+C, Ctrl+A, Ctrl+S, Ctrl+P, PrintScreen を抑止ショートカットでのコピー・保存・印刷
透かしメールアドレス＋日付の SHA-256 ハッシュを歌詞背面に 5% 不透明で繰り返し描画流出元の特定・心理的抑止
難読化歌詞テキストを 1 文字 1 span に分割。DOM 上で連続文字列として取れない単純な DOM スクレイピング

追加機能

  • DevTools 検知: window.outerHeight - window.innerHeight が 160px を超えたら歌詞を blur。
  • PrintScreen 検知: Windows で PrintScreen キーを検知したら歌詞を 3 秒間 blur。同時にクリップボードのクリアを試行。

やらないこと（v4 では見送り）

  • Canvas / SVG レンダリング: 選択・コピーを完全に防げるが、アクセシビリティと検索性が落ちる。
  • 歌詞の画像化: スクレイピングに弱く、OCR で簡単に抜ける。
  • DRM（Widevine など）: 歌詞テキストには過剰。音声側も現時点では導入しない。

流出時の対応

  1. 流出している歌詞テキスト上の透かしハッシュを画像解析ツールで読み取る
  2. 内部マッピングテーブル（SHA-256 ハッシュ → 会員 email）で逆引き
  3. Cognito 側で当該ユーザーを AdminDisableUser で停止
  4. lyrics-guard.js のアップデートを検討

抽選アルゴリズム（pickFrame）

function pickFrame(track, user, today, history) {
  // Priority 1: 記念日 (強制発動、SVG+Bannerあり)
  if (isUserBirthday(user, today))    return 'O';
  if (isUserAnniversary(user, today)) return 'P';

  // Priority 2: レア独立抽選 (Q→Rの順)
  // 結果分布: Q=1.00%, R=0.99×0.05=4.95%, common=94.05%
  if (Math.random() < 0.01) return 'Q';  // 1% Gold
  if (Math.random() < 0.05) return 'R';  // 5% Silver (Q未当選後)

  // Priority 3: 通常抽選 (アーティストプール + アクティブな季節枠)
  let pool = [...ARTIST_FRAMES[track.artist]];
  pool = pool.concat(activeSeasonalFrames(today));

  // Priority 4: 直前と違う枠を選ぶ (history は曲ごと)
  const prev = history.lastFrameOf(user, track);
  if (pool.includes(prev) && pool.length > 1) {
    pool = pool.filter(f => f !== prev);
  }

  return pool[Math.floor(Math.random() * pool.length)];
}

保護機構の継続性（lyrics-guard.js との連携）

  • v4.2 で歌詞が「画像」から「Web フォント直描画テキスト」になったため、テキスト選択・コピー・スクレイピングのリスクは画像時代より上がる。lyrics-guard.js の 5 層保護を必ず適用。

  • 1 文字 1 span 難読化は v4.2 でも継続。HTML 出力時に各文字を <span> で囲み、innerText を取られても文字順が壊れる構造を維持。

  • 装飾 SVG（K〜R の柄、紙吹雪、星）は aria-hidden="true" または CSS background-image で実装し、スクリーンリーダーから隠す。

🟢 v4.2 拡張: レター（手紙）メール送信機能

目的: フル試聴をアンロックした会員に対して、その曲の歌詞便箋（v4.2 仕様、Web フォント直描画 + ランダム枠）を、登録メールアドレス宛にレターとして送信する。会員体験の特別感を高める。v4.2 では枠／closing の抽選を server-side で実行することで、リロードによる再抽選やレア枠の不正取得を防ぐ。

実装に必要な要素

  • メール配信インフラ: Resend を推奨（Cloudflare Workers と整合、ダッシュボードが見やすい、無料枠十分）。AWS SES は次点（Cognito 同アカウント、ドメイン認証要）。最終決定は次セッション。
  • 送信用バックエンド: Cloudflare Workers で抽選 + Resend API 叩く。フロント JS だけでは送信不可。
  • HTML メールテンプレート: 便箋デザインを HTML 化、すべての CSS をインライン化、SVG は data URL で background-image に埋め込み（外部画像参照禁止: メーラーの画像ブロックで装飾が消える）、幅 680px 固定、Web View リンクを末尾に必ず付与。
  • 会員ニックネーム: Cognito の custom:nickname として保存。空ならデフォルト listener。
  • 誕生日／登録日: Cognito の custom:birthday（YYYY-MM-DD）、custom:registeredAt（ISO8601）。記念日枠 O / P の判定に使用。
  • 受信履歴: custom:letterHistory に JSON 文字列で蓄積。Cognito custom 属性は 2048 バイト上限のため、件数増加時は DynamoDB 移行（§22 参照）。

送信フロー

  1. 会員がフル試聴をアンロック（コイン消費、§9 参照）
  2. 歌詞便箋が表示される（client-side では artist プールの中で適当な枠を仮表示）
  3. 便箋下部に「📮 この手紙をメールで受け取る」ボタンが表示
  4. 会員がボタン押下 → 確認モーダル: 「このメールアドレス [user@example.com] にレターを送ります。よろしいですか?」（ニックネーム未設定なら入力欄）
  5. 会員確認 → server-side（Cloudflare Worker）で抽選実行（pickFrame + pickClosing）→ メール本文 HTML 生成
  6. Resend API でメール送信
  7. 送信成功 → letterHistory に追加記録 → 完了画面（ボタンが「📨 受け取り済み」に変わり以後グレーアウト）

21セキュリティ監査・運用ルール

静的サイトの構造的制約

前提: TAMSIC は GitHub Pages 上の完全静的サイトのため、サーバー側アクセス制御が存在しない。repo にコミット済みの MP3 は URL を知っている人なら誰でもアクセスできる。これは仕様上の制約であり、対応策は運用側で決めている。

対応方針（v4 採用）

以下の 3 層防御で緩和している:

  1. 未公開 MP3 は repo にコミットしない（Vault で保管）— 404 で防御
  2. サイト側で release-control.js + getTrackAudioSrc() の二重ゲート— UI からは locked/preview 曲を再生不可能
  3. GitHub Actions が公開日に自動配置— 人間の手動作業を排除し運用ミスを減らす
