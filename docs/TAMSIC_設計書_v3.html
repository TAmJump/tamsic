01v3 の変更サマリ
v3 の方針：TAMSIC を「かっこいい展示サイト」から「稼ぐ構造を持った会員制サイト」に転換する。デザインは一切崩さず、導線・状態制御・認証の 3 層のみ再設計する。

v2 → v3 の差分

課金タイミングどこかにボタンがあるフル試聴ボタンを押した瞬間にコイン確認→不足なら即購入導線

現状の問題（ユーザー起点）

ISSUE 01
登録する理由が弱い
「会員登録できます」の説明はあるが、触る前に登録させる動機が立ち上がっていない。
ISSUE 02
課金トリガーがない
フル試聴ボタンを押した瞬間に「買う」判断をさせていない。不足時にようやく誘導。
ISSUE 03
無料で満足できる
サンプル区間（30秒）の選び方によっては曲の核が伝わりきる。曲ごとに「Bメロ直前で止める」切り方の最適化が必要。
ISSUE 04
Cognito Hosted UI が別世界
ログイン時に AWS ドメインの白背景フォームへ飛ぶ。ブランドが一瞬で切れる。
ISSUE 05
歌詞が無防備
コピー・右クリック・スクショのいずれもできる。無料で持ち出せてしまう。
03課金導線設計（収益化の核）

基本思想

説明ではなく欲求で動かす。「なぜ会員か」「なぜ課金か」を文章で説明するのではなく、触ろうとした瞬間に壁を置く。その壁が「欲しい」を立ち上げる。TAMSIC では会員登録＝入口の壁／コイン課金＝体験の拡張、という二層で設計する。

正解のユーザー動線

01 見る
曲一覧・カバーが目に入る
モザイク・ロックアイコンで「今触れない何か」を認識させる。
02 気になる
再生ボタンを押す
未公開 or 先行曲 → 即座に「会員限定です」ポップアップ。
03 登録
TAMSIC 内でログイン／登録
AWS ドメインに飛ばず、同じページ内のフォームで完結。
04 サンプル解放
30 秒サンプルが聴ける
全トラック既に30秒運用。「もう少し」で止める切り方を曲ごとに詰める。
05 フル欲求
「フル試聴」ボタン（コイン数表示）
押す前からコイン数と残高が見える。不足ならその場で購入導線。
06 コイン購入
My Page の購入パネル
Square 外部リンク → 戻って「反映する」→ フル解放。
07 継続
次の先行曲を待つ
My Page が「次に何が来るか」を見せる場所になる。

核となる一行

設計原則 ユーザーが「触れそうで触れない」状態を作れているか？ ── これを全ページで成立させられているかを必ずレビューすること。

05会員登録導線（壁の置き方）

原則

  • ページ上部に「会員登録はこちら」を置かない。必要性が立ち上がる前の登録 CTA はノイズ。
  • 壁は「触った瞬間」にだけ現れる。何も触っていない時はブランドが主役。
  • 壁は閉じるのではなく、解放の道筋を見せる。「閉じてます」ではなく「ログインすれば聴ける」。

登録動線が出る 4 ポイント

発火ポイント出すもの
非会員が未公開曲の再生 / フルを押したモーダル（§ 04 文言①② → ログイン／登録ボタン）
非会員がフル試聴ボタンを押した（公開済）モーダル（§ 04 文言③ → ログイン／登録ボタン）
非会員が My Page を開いたカード：「My Page は会員のラウンジです」→ログイン／登録
Cognito のセッション切れモーダル：「再ログインしてください」→同じログインフォーム

07インライン認証（Cognito 直結）
v2 の問題：cognitoLogin() が https://{domain}/oauth2/authorize へリダイレクトする。その先は AWS Cognito Hosted UI で、白背景・汎用的なフォーム。TAMSIC の世界観が切れる。

v3 の方針

amazon-cognito-identity-js SDK を使い、TAMSIC ドメイン内で認証フォームを完結させる。リダイレクトはしない。Cognito の authenticateUser()（SRP 認証）で JWT を受け取り、既存の localStorage 構造に保存する。

必要な Cognito 側の設定確認

設定項目必要値確認場所
App Client の認証フローALLOW_USER_SRP_AUTH を有効化Cognito → User pool → App クライアント → 認証フロー
App Client シークレットなし（public client）※ シークレットありだとブラウザから使えない
E メール属性Alias としてサインインに使えるCognito → Sign-in experience
E メール検証有効（確認コード送信）Cognito → Message customizations
パスワードポリシー8文字以上・大文字・小文字・数字・記号の4種すべて必須（Cognitoデフォルト）Cognito → Sign-in experience → Password policy
2026.04.22 発生事案: 初回リリース時、signup.html / forgot-password.html のパスワードヒント文言・エラー文言が「8文字以上・大文字・小文字・数字」のみで記号を要求する記述が抜けていた。ユーザーが記号なしパスワードで繰り返し失敗する事故が発生。即日修正済み（当該コミット参照）。新規で認証系画面を作る際は、必ず Cognito 側の実際のパスワードポリシー（上表）と文言を一致させること。

auth.js に追加する関数

// ─── v3 追加: SRP 認証でサインイン ───
async function signInWithPassword(email, password) {
  const pool = new AmazonCognitoIdentity.CognitoUserPool({
    UserPoolId: AUTH_CONFIG.userPoolId,
    ClientId:   AUTH_CONFIG.clientId
  });
  const user = new AmazonCognitoIdentity.CognitoUser({ Username: email, Pool: pool });
  const auth = new AmazonCognitoIdentity.AuthenticationDetails({
    Username: email, Password: password
  });
  return new Promise((resolve, reject) => {
    user.authenticateUser(auth, {
      onSuccess: (session) => {
        _saveTokens({
          access_token:  session.getAccessToken().getJwtToken(),
          id_token:      session.getIdToken().getJwtToken(),
          refresh_token: session.getRefreshToken().getToken(),
          expires_in:    3600
        });
        _syncWalletFromCognito();
        resolve();
      },
      onFailure: reject,
      newPasswordRequired: () => reject(new Error('NEW_PASSWORD_REQUIRED'))
    });
  });
}

フォールバック（SRP が無効なとき）

Cognito App Client で ALLOW_USER_SRP_AUTH が有効になっていない場合、v3 のインラインフォームは機能しない。その場合は既存の cognitoLogin()（Hosted UI リダイレクト）を「詳細ログイン」として小さなリンクで残しておく。
08歌詞保護モジュール
前提：Web 上で「絶対に複製不可」は技術的に不可能。OS レベルのスクリーンショット、外部カメラ撮影は防げない。v3 の目的は「容易には持ち出せない」水準まで持ち上げることと、流出時の追跡可能性を担保すること。

保護の 5 層

層技法防げる範囲
CSSuser-select:none, -webkit-touch-callout:none, -webkit-user-drag:noneテキスト選択・モバイル長押し・ドラッグ
イベントcontextmenu / copy / cut / selectstart / dragstart を preventDefault右クリックメニュー・コピー・ドラッグ
キーCtrl/⌘+C, Ctrl+A, Ctrl+S, Ctrl+P, PrintScreen を抑止（歌詞フォーカス時のみ）ショートカットでのコピー・保存・印刷
透かしログイン中のメールアドレス（ハッシュ）を歌詞背面に 5% 不透明で繰り返し描画流出元の特定・心理的抑止
難読化歌詞テキストを innerHTML ではなく JS で 1 文字ずつ span に注入。DOM 上で連続文字列として取れない単純な DOM スクレイピング

lyrics-guard.js の API

// グローバルに 1 関数だけ公開
TAMSICLyrics.protect(containerEl, rawLyricsText, options);

// options:
// {
//   watermark: 'user@example.com',  // 透かしに入れる文字列（メールのハッシュを推奨）
//   obfuscate: true,                 // 1文字1spanに分割（既定 true）
//   onPrintScreen: 'blur'            // PrintScreen 検知時の挙動（'blur' | 'hide'）
// }

既存の nono.html / kiki.html からの置換

// 現在の innerHTML による歌詞レンダリングを置き換える
// Before:

// After:

  watermark: getUserInfo()?.email || 'guest',
  obfuscate: true

実装すべきこと（チェックリスト）

  • CSS: .full-lyrics に user-select:none; -webkit-user-select:none; -webkit-touch-callout:none; を追加
  • イベント: 歌詞コンテナに対して contextmenu / copy / cut / selectstart を preventDefault
  • 透かし: ::before 疑似要素で SHA-256 ハッシュ文字列を repeating-background として配置（5% 不透明）
  • DevTools 検知: window.outerHeight - window.innerHeight が一定値を超えた時に歌詞を非表示（鉄壁ではないが抑止）
  • PrintScreen 検知: keydown で PrintScreen キーを受けたら歌詞を 3 秒間 blur

やらないこと（v3 では見送り）

  • Canvas / SVG レンダリング:選択・コピーは完全に防げるが、アクセシビリティと検索性が大きく落ちる。v3 では DOM+CSS で妥協。
  • 歌詞の画像化:スクレイピングに弱く、OCR で簡単に抜ける。ファイルサイズも増える。
  • DRM（Widevine など）:歌詞テキストには過剰。音声側も現時点では導入しない。
09My Page の再定義

思想：管理画面ではなくラウンジ

My Page は、会員になったことの意味を忘れさせない場所。残高と履歴の数字板ではなく、「今あなたが一足先に聴ける曲」をまず見せる場所に変える。

v3 の My Page セクション構成（上から順）

11他サービスとの統一事項
TAmJump / TAmJ.Corp が運営する他サービス（Touchvoo・Medvoo・Adavoo）と、運用ルールを揃える。業種は違うが、ログイン・課金・パスワード管理・設計書フォーマットは統一することで引き継ぎコストが下がる。

Touchvoo からの移植項目

項目Touchvoo の実装TAMSIC v3 での扱い
パスワードハッシュSHA-256 ハッシュ化して D1 に保存Cognito 管理のためハッシュ化は不要。TAMSIC 側で生パスワードを保持しないのは共通
パスワードリセット管理者から再発行可能（account-master）Cognito の forgotPassword 機能を使う。管理者からも AWS コンソールで一時パスワード発行可
Square 決済リンク月額サブスク型（¥200/月）都度課金型（coin パック）のまま。Webhook の将来実装パターンだけ流用する
設計書フォーマットHTML 1 枚・badge / hint / table / code ブロックで構成TAMSIC は v2 のスタイルを維持（より引き締まった配色）
引き継ぎ方針「本設計書 1 つで 100% 引き継ぎ可能」TAMSIC v3 も同方針。v2 への参照は許容（§ 14 に集約）

統一できない項目（あえて分ける）

項目理由
データベース層Touchvoo は Cloudflare D1 / TAMSIC は Cognito カスタム属性＋ localStorage。サービス規模・データ構造が違うため統一不要。
アクセス制御Touchvoo は会社＋事業所＋ユーザーの階層 / TAMSIC は単純な会員フラグのみ。
ブランド体験Touchvoo は業務ツール（オレンジ・機能密度重視） / TAMSIC は音楽レーベル（ゴールド・余白重視）。ここを混ぜると両方が壊れる。

13セキュリティ・運用ノート
2026.04.22 発生事案：引き継ぎドキュメントに GitHub Personal Access Token（PAT）の平文スクリーンショットが含まれ、チャット経由で共有された。当該 PAT は即時失効済み / 新 PAT 発行済み。以降、同じ事故を起こさないため以下を運用ルールとする。

認証情報の取り扱い（厳守）

対象正しい扱いやってはいけない
GitHub PAT1Password / Bitwarden 等に保存。期限切れ前に再発行スクショ・画像・チャット・メール本文に貼り付け
Cognito ユーザーのパスワードCognito が SHA-256 でハッシュ保存。誰も復元できないのが正常「今登録されている PW を確認したい」は不可。リセットしか手段がない
AWS アクセスキー最小権限 IAM ロール＋ローテーションソースコードに直書き
Square API トークンCloudflare Workers の環境変数 / AWS Secrets ManagerHTML / JS にベタ書き

パスワードリセットの手順（管理者向け）

  1. AWS コンソール → Cognito → User pool - kzohfe → ユーザー
  2. 対象ユーザーを選択 → 「アクション」→「パスワードをリセット」
  3. 一時パスワードが発行される（メール送信）
  4. ユーザーが次回ログイン時に新しいパスワードを設定
ユーザーからの問い合わせフロー：「パスワードを忘れた」→ TAMSIC の login.html にある「パスワードを忘れた方」→ forgot-password.html でメール入力 → 確認コード → 新パスワード設定。管理者が介入する必要はない。

歌詞流出が判明した場合

  1. 流出している歌詞テキスト上の透かしハッシュを確認（§ 08）
  2. そのハッシュから会員 ID を逆引き（内部マッピングテーブル）
  3. Cognito 側で当該ユーザーを無効化（AdminDisableUser）
  4. 流出経路の評価と lyrics-guard.js のアップデート検討
